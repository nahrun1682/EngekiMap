# スキルを書く

## 概要

スキル作成は「プロセス文書に対する TDD」。

手順:
1. 圧力シナリオを作成して baseline 失敗を観測（RED）
2. その失敗に効く最小スキルを書く（GREEN）
3. 新しい抜け道を塞いで再検証（REFACTOR）

**中核原則:** スキルなし状態で失敗を観測していなければ、スキルの有効性は証明できない。

## スキルとは

スキルは、再利用可能な技法・パターン・ツールの参照ガイド。

**Skills are:**
- 再利用できる手法
- 問題解決パターン
- ツール操作ガイド
- 参照資料

**Skills are NOT:**
- 一度限りの体験談

## いつ作るか

作る:
- 再利用性がある
- 他者にも有用
- 技術判断を伴う

作らない:
- 一回限りの対応
- 既に標準化されている内容
- プロジェクト固有規約（CLAUDE.md へ）
- 機械的に自動検証できるルール

## ファイル構造

```text
skills/
  skill-name/
    SKILL.md
    supporting-file.*
```

分離ファイルは「重い参照」か「再利用ツール」のみ。

## SKILL.md 規約

Frontmatter:
- `name` と `description` のみ
- 合計1024文字以内
- `name` は英数字とハイフン
- `description` は三人称・トリガー条件のみ（プロセス要約禁止）

## CSO（検索最適化）

将来の Claude が見つけやすくする:
- description は "Use when..." 開始
- 症状語・エラー語・ツール名を含める
- 技術非依存トリガーを優先
- 名前は動詞中心で意味を明確化

## トークン効率

- 頻出スキルは短く保つ
- 詳細フラグは `--help` 参照へ逃がす
- 重複説明は cross-reference
- 例は少数精鋭

## 他スキル参照ルール

- `**REQUIRED SUB-SKILL:** Use superpowers:...` 形式を使う
- パス直書きや `@.../SKILL.md` 強制ロードは避ける

## Flowchart ルール

使う:
- 非自明な分岐
- ループ判断
- A/B 選定

使わない:
- 線形手順
- 参照一覧
- コード例

## 反パターン

- 体験談ベースの記述
- 多言語に分散した薄いサンプル
- 意味のないラベル（step1/helper2）
- フローチャート内にコードを書く

## 鉄則

```
NO SKILL WITHOUT A FAILING TEST FIRST
```

新規作成だけでなく既存編集にも適用。

## スキル種別別テスト

### Discipline-Enforcing
規律系（TDD等）:
- 圧力下での逸脱有無を確認

### Technique
手順系:
- 新規シナリオへ適用できるか確認

### Pattern
思考モデル系:
- 適用すべき場面の判定精度を確認

### Reference
参照系:
- 情報検索と適用の正確さを確認

## RED-GREEN-REFACTOR 運用

### RED
- スキルなしで圧力シナリオ実行
- 合理化発言を逐語記録

### GREEN
- その合理化に効く最小スキルを書く
- 同一シナリオで準拠を確認

### REFACTOR
- 新しい合理化を検出
- 明示カウンターを追加
- 再検証して抜け道を塞ぐ

## チェックリスト

- [ ] baseline 失敗を記録した
- [ ] description が trigger-only になっている
- [ ] name 規約を満たしている
- [ ] 検索語（症状/エラー/ツール）を含む
- [ ] 最小スキルで baseline 問題を解消した
- [ ] 再テストで準拠を確認した
- [ ] 新しい抜け道への対策を追加した
- [ ] git へコミットした

## 結論

スキル作成は、コードと同じく TDD で品質を担保する。
失敗観測なしのスキルは未検証コードと同じ扱い。
